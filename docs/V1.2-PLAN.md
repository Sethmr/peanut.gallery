## 1. Goals, non-goals, done-definition

### Goals
1. Surface the Director's decision-making as **structured logs** and a **live debug panel** in the side panel.
2. Retune the cascade spacing so fast exchanges feel conversational, not staged.
3. Lock `lib/director.ts` under **fixture-driven tests** covering scoring, cooldowns, cascade probability, and the v1.1.1 silence path.
4. Wire a **pre-merge gate** (`typecheck + lint + test-director`) so silent-failure regressions stop slipping past types.

### Non-goals
- Smart Director v2 (LLM-assisted routing) — that's v1.4.0. Keep the rule-based scorer intact.
- Persona-pack refactor — that's v1.3.0. Don't touch `lib/personas.ts` shape.
- Any backend wire-protocol change beyond the new **optional** SSE event. Older extensions ignoring the new event must keep working unchanged.
- No CI setup yet (pre-merge gate is local-only via git hook). CI is a post-v1.2 chore.

### Done-definition (every box must be ticked to call v1.2 shipped)
- [ ] `director_decision` log line includes `{ ts, pick, score, top3, cascadeLen, cooldownsMs, reason }` at `info+`.
- [ ] New SSE event type `director_decision` emitted from the transcribe route, consumed by the side panel.
- [ ] Long-pressing the side-panel version badge reveals a collapsible debug panel showing the last N decisions as a mini-table.
- [ ] Older extensions (≤ v1.1.2) still connect to a v1.2 backend without errors (forward-compat).
- [ ] Cascade-delay tuning bakes new numbers chosen from at least 2 real-session log captures.
- [ ] `scripts/test-director.ts` runs green against the fixture set and catches regressions for ISSUE-004, ISSUE-006, and the v1.1.1 silence path.
- [ ] `npm run typecheck && npm run lint && npm run test:director` all pass.
- [ ] `.husky/pre-commit` (or equivalent) runs the gate automatically on every commit.
- [ ] `CHANGELOG.md` Unreleased → `[1.2.0] — YYYY-MM-DD`.

## 3. Phase 1 — Enrich the director_decision log + add the SSE event

**Why this is first:** the debug panel (Phase 2) and the tests (Phase 4) both consume this shape. Lock the schema now.

### 3.1 Decision log schema

Target shape (camelCase, JSON-serializable only):

```ts
interface DirectorDecisionLog {
  ts: string;              // ISO 8601, e.g. "2026-04-17T23:41:02.814Z"
  sessionId: string;
  pick: string;            // primary persona id
  score: number;           // primary's final score
  top3: Array<{            // ordered; always includes pick at [0]
    id: string;
    score: number;
  }>;
  chain: string[];         // full cascade chain (pick first)
  delays: number[];        // ms before each persona fires, parallel to chain
  cascadeLen: number;      // chain.length
  cooldownsMs: Record<string, number>; // per-persona ms since last fire, for all 4
  reason: string;
  isSilence: boolean;
  isForceReact: boolean;
}
```

### 3.2 Files to edit

- `lib/director.ts`
  - Extend `TriggerDecision` to carry `{ top3, cooldownsMs, score }` alongside the existing fields. Keep `personaIds`, `reason`, `delays` as-is — downstream code reads them.
  - Inside `decide()`, after sorting scores, build the full shape and return it.
  - Replace the existing `logPipeline({ event: "director_decision", ... })` block with a log emitting the full schema above.
  - `cooldownsMs` comes from `this.cyclesSinceFire` × the director's trigger interval; expose it as a computed field at decision time. If the trigger interval isn't available inside the Director class, compute cooldowns in the route (see 3.3) and pass them into the log there instead.

- `app/api/transcribe/route.ts`
  - Remove the duplicate `log.info("director_decision", ...)` at ~line 387 — it's redundant now that the Director emits the full shape.
  - Keep the SSE `send("director_decision", payload)` in the same block, emitting the identical payload the Director just logged. Reuse the existing `send()` helper.
  - Include `sessionId` on the payload explicitly (don't rely on a header fallback) so the debug panel can filter cleanly when multiple sessions overlap during dev.
  - Wrap the emit in a `try { send(...) } catch {}` so a closed stream never crashes the director path.

### 3.3 Back-compat

The Director decision log already exists at `info` level; this phase enriches it, not replaces it. Field additions are additive. **Do not rename `chain`, `reason`, `cascadeCount`, or `isSilence`** — external log consumers may already key on them. `cascadeCount` should continue to equal `chain.length` (compute it rather than storing redundantly, but preserve the field).

### 3.4 Acceptance criteria

- [ ] Run `DEBUG_PIPELINE=true npm run dev`, open a YouTube URL via the web app or the extension, trigger at least 3 director cycles (one content-driven, one silence, one force-react).
- [ ] `tail -5 logs/pipeline-debug.jsonl | jq 'select(.event=="director_decision")'` shows the full schema every time.
- [ ] `curl -N` against `/api/transcribe` (or open the Network tab and watch the SSE stream) shows a `director_decision` event with the same payload, once per cycle.
- [ ] Connect an unchanged v1.1.2 extension against the v1.2 backend — persona messages still render, no console errors about unknown events.

### 3.5 Risks

| Risk | Mitigation |
|---|---|
| Payload size bloat — top3 + cooldowns could add ~250 bytes per decision | Trivial vs. transcript chunk size; not worth worrying about. |
| SSE event name collision — we already emit `director_status` or similar? | Grep `app/api/transcribe/route.ts` for `send(` calls; confirm `director_decision` isn't in use before wiring. If it is, rename to `director_trace`. |
| Log consumers break on renamed fields | Don't rename. Only add. |

## 4. Phase 2 — Side-panel debug panel

**Why this is second:** Phase 1 emits the data. Phase 2 renders it. The panel is the single most valuable tuning tool for Phase 3.

### 4.1 UX shape

- **Entry:** long-press (750ms) on the version badge in the side-panel footer. No visible affordance on short press — keeps the panel out of the way for normal users.
- **Panel:** collapsible section at the bottom of the side panel, above the footer. Header reads `Director trace · last 20`. Inside is a vertical scroll-list of rows, newest at the top.
- **Row layout** (one row per decision, max 3 lines tall, monospace-ish):
  - Line 1: `🎭 pick · score (top3 short form) · cascadeLen`
  - Line 2: transcript preview (first 80 chars of whatever the Director scored) in muted grey
  - Line 3: `reason` string, slightly indented
- **Clearable:** small `Clear` button in the panel header resets the buffer.
- **Persistence:** panel open/closed state persisted to `chrome.storage.local` under `debugPanelOpen`. Buffer is in-memory only — flushes on side-panel close.
- **Off by default** for new installs. `debugPanelOpen: false` is the default.

### 4.2 Files to edit

- `extension/sidepanel.html`
  - Add a new `<section id="directorTrace" hidden>` block with the scroll container and header. Place it above the existing footer.
  - Style with Tailwind-esque inline classes consistent with the rest of the sidepanel. Dark background, accent border colored by persona (reuse the `--persona-*` CSS vars if present, otherwise hard-code the 4 persona colors from `docs/CONTEXT.md §UI Polish`).

- `extension/sidepanel.js`
  - **Long-press handler:** add `pointerdown`/`pointerup` listeners on the version badge element. Kick a 750ms timer on down; if it completes without an up, toggle `debugPanelOpen`, render, and re-save to `chrome.storage.local`.
  - **SSE consumption:** in the existing SSE event router (the one that already handles `persona_fire`, `transcript`, etc.), add a case for `director_decision`. Push the payload onto a bounded ring buffer (`const MAX_TRACE = 20;`).
  - **Render:** `renderDirectorTrace()` rebuilds the list from the buffer on every push. Use `document.createDocumentFragment()` to avoid repainting 20 nodes on every event.
  - **Clear:** button hook that empties the ring buffer and re-renders.
  - **Forward-compat handling:** if the panel is closed, still push to the ring buffer silently so opening it shows history.

- `extension/sidepanel.js` (guard): Wrap the new code in feature-detection — if the incoming event doesn't have `top3`, skip the row render and fall back to a minimal line (`pick · cascadeLen`). This way even if someone pins an older backend it won't throw.

### 4.3 Acceptance criteria

- [ ] Long-pressing the version badge toggles the panel open; short-pressing does nothing.
- [ ] Panel is closed by default on a fresh install.
- [ ] Open state persists across side-panel close/reopen.
- [ ] During an active session the panel populates in realtime, newest entries on top, capped at 20.
- [ ] `Clear` empties the list.
- [ ] Disconnect the backend mid-session — panel shows the events that arrived before disconnect, no console errors.
- [ ] Against an older backend that emits a minimal payload, the panel renders fallback rows without throwing.

### 4.4 Risks

| Risk | Mitigation |
|---|---|
| Long-press conflicts with text-select on the badge | Badge is a `<span>` with `user-select: none` — verify in the existing CSS; add if missing. |
| Side panel freezes on long sessions (unbounded render work) | Ring buffer is capped at 20; re-render is O(N) with N≤20. Fine. |
| Chrome storage quota | Only `debugPanelOpen: boolean` persisted. Zero risk. |

## 5. Phase 3 — Cascade-delay retune (measure first, tune second)

**Why this is third:** Phases 1–2 give us the observability to measure. Don't tune blind.

### 5.1 Measurement

1. Run **two** real-session captures against `peanutgallery.live`:
   - **Fast exchange fixture:** a TWiST clip with overlapping speakers / rapid topic changes. 3–5 minutes.
   - **Slow exchange fixture:** a solo segment (monologue or interview with long answers). 3–5 minutes.
2. Capture `logs/pipeline-debug.jsonl` for each.
3. Aggregate: for each cascade event, compute `max(delays)` (time from primary fire to last cascade member). Plot or eyeball the distribution.

### 5.2 Target numbers (starting hypothesis — tune from data, not this doc)

Current: `CASCADE_DELAY_MIN_MS = 2000`, `CASCADE_DELAY_MAX_MS = 4000`.

Hypothesis: during fast exchanges the floor of 2s is the problem, not the ceiling. Try:

- `CASCADE_DELAY_MIN_MS = 900`
- `CASCADE_DELAY_MAX_MS = 3200`
- Optionally, **context-aware jitter**: shrink the window when the Director sees ≥ 2 pattern hits in the last 3 cycles (i.e., a "hot" segment). That adds one flag to the Director's state — worth trying only if the uniform retune doesn't feel right.

### 5.3 Files to edit

- `lib/director.ts`
  - Adjust the two constants.
  - If going with context-aware jitter: add a `hotStreak: number` field to the class, increment on cycles with ≥ 2 non-zero scores, decay otherwise, and use it to scale the jitter window.

### 5.4 Acceptance criteria

- [ ] Before/after log captures show median `max(delays)` dropping on fast fixtures and staying flat on slow fixtures.
- [ ] Subjective: watch 5 minutes of live feedback with the debug panel open. Cascades feel like conversation, not like a game show buzzer.
- [ ] No test in Phase 4 needs to be loosened to accommodate the new numbers — if it does, the tuning overshot.

### 5.5 Risks

| Risk | Mitigation |
|---|---|
| Personas start talking over each other in the UI | Personas render in their own columns; overlap is fine visually. If streamed tokens collide in time, that's actually the show we want. |
| Tighter delays expose a race in `personaEngine.fireSingle` | Rare. Mitigation: if seen, add a 100ms floor before each fire as a safety; don't undo the retune. |

## 6. Phase 4 — Director tests + fixtures

**Why this is fourth:** Phases 1 and 3 change Director outputs. Tests guard the new shape and the new tuning.

### 6.1 Structure

- **Location:** `scripts/test-director.ts` (new). Mirror the shape of `scripts/test-personas.ts`.
- **Runner:** `tsx scripts/test-director.ts` — exits 0 on success, non-zero on any fixture failure. Prints a concise pass/fail summary.
- **Fixtures:** `scripts/fixtures/director/*.json`. Each fixture describes an input + expected outputs.

### 6.2 Fixture schema

```ts
interface DirectorFixture {
  name: string;                    // short identifier
  description: string;
  initialState?: {                 // optional: seed director state
    recentFirings?: string[];
    cyclesSinceFire?: Record<string, number>;
  };
  input: {
    transcript: string;
    isSilence?: boolean;
  };
  assertions: {
    pickIn?: string[];             // pick must be one of these
    pickNot?: string[];            // pick must NOT be any of these
    chainMaxLength?: number;
    chainMinLength?: number;
    reasonContains?: string;
    cascadeLenExact?: number;
    mustFirePersona?: string;      // persona that must appear in chain
    mustNotFire?: string;          // persona that must not appear
  };
  runs?: number;                   // repeat with different RNG seeds (default 50); all runs must satisfy assertions
}
```

**Randomness:** the Director uses `Math.random()` in several places. Tests run each fixture N times (default 50) and assert on distribution — e.g., `pickIn` must hold for ≥ 95% of runs, `mustFirePersona` must hold for ≥ 80%. Exact thresholds per fixture.

### 6.3 Required fixtures (minimum viable set)

1. **`baba-booey-fact-driven.json`** — transcript with a clear factual claim; Baba Booey must be pick in ≥ 90% of runs.
2. **`troll-hype-speech.json`** — transcript full of buzzwords; Troll must be pick in ≥ 85% of runs.
3. **`fred-mood-shift.json`** — awkward pause / topic pivot; Fred should fire in the chain in ≥ 70% of runs.
4. **`jackie-setup-available.json`** — comparison/absurdity language; Jackie should fire in chain in ≥ 70% of runs.
5. **`silence-tick.json`** — `isSilence: true`, transcript empty; chain length must be ≤ 2 in 100% of runs (v1.1.1 invariant).
6. **`force-react-regression.json`** — mimics ISSUE-004: transcript present, director called, force-react flag set. Not actually Director's job any more but serves as a "this shape is preserved" guard — assertion is that `decide()` still returns the full shape without throwing.
7. **`pause-repeat-regression.json`** — mimics ISSUE-006: session paused, called twice. Director must produce different chains across the two runs (no forced repetition). (If the regression really lives in the route not the Director, move this fixture to a separate route-level test.)
8. **`dry-spell-forces-fire.json`** — seed `cyclesSinceFire.joker: 12`, feed a neutral transcript; Jackie must fire in chain in ≥ 90% of runs (dry-spell boost caps at 6 but should dominate neutral scoring).
9. **`recency-penalty.json`** — seed `recentFirings: ['troll','troll','troll','troll']`, feed a troll-magnet transcript; Troll should fire in ≤ 40% of runs despite pattern match.
10. **`cascade-delay-monotonic.json`** — any fixture; assert `delays` is monotonically non-decreasing (structural invariant).

### 6.4 Files

- `scripts/test-director.ts` (new)
- `scripts/fixtures/director/` (new dir, 10 JSON files above)
- `package.json` — add script: `"test:director": "tsx scripts/test-director.ts"`

### 6.5 Acceptance criteria

- [ ] `npm run test:director` exits 0 on current main after Phase 1–3 land.
- [ ] Deliberately revert the Phase 3 cascade tuning on a branch: tests still pass (they don't assert on magnitudes, only structure + distributions).
- [ ] Deliberately revert the silence-cap in `decide()` (`if (isSilence && chain.length > 2) ...`): the `silence-tick.json` fixture fails loudly.

### 6.6 Risks

| Risk | Mitigation |
|---|---|
| Flaky tests from tight probability thresholds | Use 50-run default and generous thresholds (95/85/70 per fixture above). If any fixture flakes, raise runs to 100 before loosening thresholds. |
| Test-only hooks leak into production code paths | Keep the Director's public API unchanged; expose new fields on `TriggerDecision` (readonly). No "test mode" flag. |
| Fixtures grow stale when personas tune further | Acceptable cost — tests catching persona retunes is the point. Update fixtures in the same PR as any Director change. |

## 7. Phase 5 — Pre-merge gate

**Why this is fifth:** all the code written in Phases 1–4 is the thing we're gating. Order matters.

### 7.1 What the gate runs

```bash
npm run typecheck && npm run lint && npm run test:director
```

### 7.2 Files

- `package.json`
  - Add `"typecheck": "tsc --noEmit"` to `scripts`.
  - Add `"test:director": "tsx scripts/test-director.ts"` (already from Phase 4).
  - Add `"check": "npm run typecheck && npm run lint && npm run test:director"` as a convenience target.
  - Add `"prepare": "husky"` in `scripts` (needed for husky's install hook).

- `devDependencies`: add `"husky": "^9.1.0"` via `npm install --save-dev husky`.

- `.husky/pre-commit` (new, executable):

  ```bash
  #!/usr/bin/env sh
  npm run check
  ```

  Make sure the file has `chmod +x`. Husky v9 doesn't need the preamble line anymore.

### 7.3 Opt-out

For emergency commits, the standard `git commit --no-verify` path works. Don't add a custom bypass env var.

### 7.4 Acceptance criteria

- [ ] Fresh clone → `npm install` → `git commit` on any change runs the full gate.
- [ ] Forcing a type error in `lib/director.ts` makes `git commit` fail with a clear `tsc` error.
- [ ] Forcing a fixture assertion to fail makes `git commit` fail with the fixture name in the output.
- [ ] Lint is already green on `main` before we wire this; if it isn't, fix warnings before enabling the hook (don't let the first post-enable commit be a lint-fix chore).

### 7.5 Risks

| Risk | Mitigation |
|---|---|
| Contributors on Windows hit hook-execution issues | Husky handles this. If not, document `core.hooksPath` setup in README "Contributing". |
| The gate adds perceptible commit latency | Typecheck + lint + 50-run fixture tests should land under 15s cold, <5s warm. Acceptable. |
