> Step-by-step playbook to ship v1.3.0 "TWiST Pack" safely on top of v1.2.0 without breaking the Howard Stern pack that's in CWS review right now. Every phase is self-contained with exact file paths, code shape, acceptance criteria, rollback, and a concrete verification step that uses the v1.2 logging + test harness to prove nothing regressed. Phases are ordered so later ones depend only on earlier ones.

## 1. Goals, non-goals, done-definition

### Goals
1. Refactor personas into **selectable packs**, with Howard Stern as the default pack (zero behavior change for existing users) and TWiST as the flagship second pack.
2. Ship the **TWiST pack** — Jason Calacanis, Molly Wood, Alex Wilhelm, Lon Harris — written to the same depth and voice-fidelity as the Howard pack.
3. Add a **pack-swap dropdown** to the side panel, persisted to `chrome.storage.local` and forwarded server-side on each session start.
4. Extend the v1.2 **Director test harness** with a `--pack` flag and pack-specific fixtures, so both packs must pass the pre-commit gate.
5. Update the **debug panel** to show the active pack so tuning sessions know which pack they're watching.

### Non-goals
- **Pack-creation installer** (scaffolder CLI / Claude skill). Originally scoped in the roadmap but explicitly deferred to v1.3.1 — the MVP of v1.3 is two shipped packs + swap UI. Installer is a community-facing affordance, not a flagship feature. See §9.1.
- **Smart Director v2** (LLM-assisted routing) — that's v1.4.0.
- **Per-persona voice / TTS** — that's v1.5.0.
- Changing Director scoring thresholds, cascade probabilities, or the pace multiplier. Any tuning must be its own PR after v1.3 lands.
- New LLM provider integrations. Both packs use the existing `claude-haiku` / `groq-llama-70b` / `groq-llama-8b` trio.
- Server-side storage of pack selection. Pack lives in the client and is forwarded per-request, same wire pattern as the v1.2 rate dial.

### Done-definition (every box must be ticked to call v1.3 shipped)
- [ ] `lib/packs/howard/` and `lib/packs/twist/` exist and each export a `Pack` with 4 personas.

## 3. Architecture — the Pack abstraction

### 3.1 Fixed archetype slots (load-bearing)

Every pack provides content for the **same 4 archetype slots**, keyed by these exact IDs:

```
producer   — the fact-checker foil          (Claude Haiku)
troll      — the cynical commentator        (Groq Llama 70B)
soundfx    — the sound-effects quiet genius (Groq Llama 8B)
joker      — the comedy writer              (Claude Haiku)
```

**Why this matters:** `lib/director.ts`'s `PERSONA_PATTERNS` is keyed by these IDs. `extension/sidepanel.html`'s 4 persona cards are keyed by these IDs. `components/PersonaIcon.tsx`, `app/watch/page.tsx`, `app/page.tsx`, `app/install/page.tsx`, `docs/index.html`, and ~14 other files reference them. Changing an ID would ripple through all of them; keeping them fixed means a pack refactor is **content-only** at the boundary, which is the safest possible shape.

A pack CANNOT add a 5th persona in v1.3. Variable persona counts are a v1.4+ conversation — not this release.

### 3.2 The `Pack` type

```ts
// lib/packs/types.ts (new)
import type { Persona } from "@/lib/personas"; // keep the Persona interface where it is

export interface PackMeta {
  /** Machine id. Lowercase, url-safe, matches the folder name. */
  id: string;
  /** Human-facing name shown in the dropdown. */
  name: string;
  /** One-line description, shown as a tooltip. */
  description: string;
  /** The show / universe this pack evokes (for the debug panel header). */
  universe: string;
  /** ISO 8601 date the pack was last edited. Useful for changelog. */
  updatedAt: string;
}

export interface Pack {
  meta: PackMeta;
  /** Exactly 4 personas, in slot order: producer, troll, soundfx, joker. */
  personas: Persona[];
  /**
   * OPTIONAL — if present, the Director uses these patterns instead of the
   * default set. Unset ⇒ use default patterns (current behavior).
   * v1.3.0 ships with this UNSET on both packs; the hook is there so a future
   * non-startup pack can opt in without a second refactor.
   */
  patterns?: typeof import("@/lib/director").PERSONA_PATTERNS;
}
```

### 3.3 Layout

```
lib/
├── packs/
│   ├── types.ts                ← the Pack + PackMeta interfaces
│   ├── index.ts                ← registry: { howard, twist }, resolvePack(id)
│   ├── howard/
│   │   ├── personas.ts         ← verbatim copy of today's 4 personas
│   │   └── index.ts            ← exports Pack
│   └── twist/
│       ├── personas.ts         ← new — Calacanis / Wood / Wilhelm / Harris
│       └── index.ts            ← exports Pack
├── personas.ts                 ← shim: re-exports Howard's personas + types
│                                  so every existing `import { personas } from
│                                  "./personas"` keeps working
├── persona-engine.ts           ← takes a Pack, not a bare personas[]
└── director.ts                 ← takes optional Pack for pattern override
```

### 3.4 The loader

```ts
// lib/packs/index.ts
import { howardPack } from "./howard";
import { twistPack } from "./twist";
import type { Pack } from "./types";

export const PACKS: Record<string, Pack> = {
  howard: howardPack,
  twist: twistPack,
};

export const DEFAULT_PACK_ID = "howard";

export function resolvePack(id: string | undefined | null): Pack {
  if (!id) return PACKS[DEFAULT_PACK_ID];
  const key = String(id).toLowerCase().trim();
  return PACKS[key] ?? PACKS[DEFAULT_PACK_ID];
}

export function listPacks(): PackMeta[] {
  return Object.values(PACKS).map((p) => p.meta);
}
```

`resolvePack` NEVER throws. Unknown id → default. This is the single point where the "forward-compat" promise lives.

## 4. Phase 1 — Extract Howard into a pack (zero behavior change)

**Why this is first:** this phase proves the refactor is safe. When it's done, running `npm run check` must produce identical `director_decision` logs to the pre-v1.3 main branch. No pack UI yet, no TWiST yet, nothing user-visible.

### 4.1 Steps

1. Create `lib/packs/types.ts` (§3.2).
2. Create `lib/packs/howard/personas.ts` — copy the `personas` array from `lib/personas.ts` verbatim. **Do not edit system prompts.** They're character-authored.
3. Create `lib/packs/howard/index.ts`:
   ```ts
   import { howardPersonas } from "./personas";
   import type { Pack } from "../types";
   export const howardPack: Pack = {
     meta: {
       id: "howard",
       name: "Howard Stern Show",
       description: "Baba Booey, The Troll, Fred, Jackie — the original lineup.",
       universe: "Stern Show",
       updatedAt: "2026-04-17",
   ```

4. Create `lib/packs/index.ts` with `PACKS`, `DEFAULT_PACK_ID`, `resolvePack`, `listPacks` (§3.4). TWiST entry is a placeholder stub that re-exports `howardPack` so the loader type-checks — Phase 2 replaces the stub.
5. Rewrite `lib/personas.ts` as a shim:
   ```ts
   // Backward-compat shim. New code should import from `lib/packs/howard`.
   export { howardPersonas as personas } from "./packs/howard/personas";
   export { buildPersonaContext } from "./packs/howard/context";
   export type { Persona, OtherPersonaResponse, ConversationEntry } from "./packs/types-persona";
   ```

   Actually simpler: keep `Persona`, `buildPersonaContext`, `OtherPersonaResponse`, `ConversationEntry` in `lib/personas.ts` (they're pack-agnostic) and only move the `personas: Persona[]` array. That avoids moving interfaces around.

7. Update `app/api/transcribe/route.ts` to construct the engine with `resolvePack(pack)`.
8. Update `app/api/personas/route.ts` similarly.
9. Update `scripts/test-personas.ts` to take `--pack <id>` (default howard).
10. Do NOT touch `lib/director.ts` in this phase. Its `PERSONA_PATTERNS` stays where it is.

### 4.2 Acceptance criteria for Phase 1

## 5. Phase 2 — Author the TWiST pack

**Why this is the biggest phase:** the pack's VALUE is the character depth. The code is trivial. Budget the time on the voice work.

### 5.1 Research doc (do this BEFORE writing prompts)

Create `docs/packs/twist/RESEARCH.md` with 4 sections — one per persona. Each section has:

1. **Who they are in real life.** Biographical sketch, 3-5 sentences.
2. **Signature voice markers.** 5-10 actual phrases they use on TWiST ("that's a VAULT!", "Molly Wood over here thinks…", etc.). These are the verbal fingerprints the prompt will lean on.
3. **Taboo topics.** What the character would never joke about (personal struggles, family members off-limits, sponsor relationships).
4. **Cross-persona callbacks.** The specific dynamics between them — Jason's "Molly, what do YOU think?" turn, Alex's pushback on Jason's pronouncements, Lon's dry-aside energy.
5. **Archetype slot mapping.** Which real person maps to which slot and WHY.

|-----------|--------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------|

| troll     | **Jason Calacanis**| The boldest opinion-holder, the guy who says the spicy thing. On his own show the "dunk" energy comes mostly from him. Fan of his own takes, like The Troll.     |
| soundfx   | **Lon Harris**     | Quiet encyclopedic knowledge, deadpan asides, underrated contributions. Closest TWiST analog to Fred Norris.                                                      |

Seth confirms or reshuffles this mapping before Phase 2 prompts get written. **Blocker for Phase 2.2.**

### 5.2 Prompt-writing conventions

Each TWiST persona prompt follows the exact same structure as its Howard counterpart:
- `WHO YOU ARE` section
- `YOUR VOICE` bullets
- Verbal fingerprints list (derived from §5.1.2)
- Topic-specific behaviors (§5.1.3 taboos baked in)
- Cascade play section (§5.1.4 cross-persona callbacks)
- Anti-repetition rules (verbatim copy from Howard — this is a universal property, not pack-specific)
- Format rules (1-2 sentences, etc.)

This mirroring keeps the prompt surface area identical across packs so the Director's scoring model still produces sensible scores for either.

## 7. Phase 4 — Extend the Director test harness

### 7.1 `scripts/test-director.ts` changes

- New CLI arg: `--pack <id>`. Default: `"howard"`. If `"all"`, run both packs sequentially.
- Fixture loader filters by top-level `"pack"` field in each fixture JSON. If a fixture omits `"pack"`, it's pack-agnostic (runs for every pack).
- Existing 10 Howard fixtures: add `"pack": "howard"` to each. TWiST fixtures from §5.4 carry `"pack": "twist"`.
- Pack-agnostic fixtures (structural invariants: `cascade-delay-monotonic`, `silence-tick`, `force-react-regression`, `pause-repeat-regression`) stay pack-less so they run for every pack.
- Output: `"Director test harness — pack=<id>, N fixture(s), R runs each"` header per pack. Non-zero exit if any pack fails.

### 7.2 `package.json`

```json
"test:director": "tsx scripts/test-director.ts --pack all",
"test:director:howard": "tsx scripts/test-director.ts --pack howard",
"test:director:twist": "tsx scripts/test-director.ts --pack twist",
"check": "npm run typecheck && npm run lint && npm run test:director"
```

`test:director` now defaults to `--pack all`, so the existing pre-commit hook transparently covers both packs after this phase lands.

### 7.3 Acceptance criteria for Phase 4
- `npm run test:director` → `"Director test harness — pack=howard"` and `"Director test harness — pack=twist"` both green.
- Negative test: flip a threshold in `twist-molly-fact-check.json` to impossibility → harness fails with the per-run counts for that fixture. Revert.
- `npm run check` runs the two-pack suite end-to-end in under ~10s (harness stays fast).

### 7.4 Rollback for Phase 4
- `--pack` defaults to `"howard"` if removed, so harness falls back to the v1.2 shape. Fixtures with a `"pack"` field parse fine even if the flag isn't passed (the field is just ignored).

## 11. Safety strategy — leaning on v1.2's logging

Seth's ask: **don't break anything.** Three things make v1.3 safer than v1.2 was:

1. **The fixture harness already exists.** Every phase runs `npm run check` as its last verification. A broken Director shape, a dropped field, a regressed cascade — any of these turn the harness red. Phase 1's "zero behavior change" claim is a green harness run on the untouched Howard fixtures.

2. **The `director_decision` log is the canary.** For every manual smoke run, we diff the log lines between pre-v1.3 main and the v1.3 branch. If `pick`, `score`, `top3`, `chainIds`, `delays`, `cooldownsMs` shapes change, the diff screams. The log was explicitly built in v1.2 to enable exactly this kind of refactor.

3. **Forward-compat at every wire edge.** The server `resolvePack` ALWAYS returns Howard on missing/unknown. The extension sends `"howard"` by default. Old clients + new server = Howard. Old server + new client = Howard (server just ignores the unknown field). Any dropped route is a Howard route.

If ANY phase goes red mid-implementation, revert that phase's commit and re-plan. Don't paper over with a follow-up fix.
